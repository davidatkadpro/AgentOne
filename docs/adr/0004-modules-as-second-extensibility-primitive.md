# Modules as a second extensibility primitive

AgentOne gains a second extensibility primitive — the **Module** — alongside the existing **Skill**. A Module is a schema-owning, event-emitting, skill-bundling domain (e.g. `projects`, `email`, `proposals`, `invoicing`) living under `modules/<name>/`. It declares its database tables via versioned SQL migrations, contributes typed event variants, exposes a typed service interface, and ships the Skills that let the agent operate on its state. Skills remain the primary extensibility mechanism for *capabilities*; Modules add an extensibility mechanism for *domains*. A Module's service is callable by any in-process actor — Skill tool handlers, HTTP route handlers, hooks, schedulers, and other Modules — not only by the agent. Mutations write to an immutable `audit_log` so the chat history is no longer assumed to be the complete record of "what happened to the system."

## Considered alternatives

- **Force everything to be a Skill.** Schema, migrations, and event-type contributions get pushed into SKILL.md frontmatter; a 30-skill cluster shares an implicit schema. Rejected — Skills are intended to be small, loadable on demand, and procedural. Cramming a domain with 4 tables and 12 actions into sibling Skills hides the shared model that a contributor must understand together; the load-on-demand semantic does not apply to schema.
- **Bake new domains into Core.** A `ProjectService`, `EmailService`, etc. live in `src/`, instantiated like `WikiEngine`. Rejected because it defeats the user's stated goal of letting other operators extend or replace these domains without forking the runtime. AEC-specific schema would require a code fork.
- **Modules callable only via Skills.** Closes the agent-as-gatekeeper boundary. Rejected — forces a chat round-trip and an LLM call for routine CRUD (rendering a task list, marking a task done from the UI), which is wasteful and non-deterministic. The v2 frontend is explicitly mixed (chat alongside non-LLM user actions); a service-layer that only the agent can call cannot serve that frontend.
- **One Module ADR per concern (Module shape; open services; audit log) — three ADRs.** Rejected as over-fragmentation. The three decisions are tightly coupled — open services without an audit log breaks the v1 assumption that conversation history records every mutation; an audit log without open services is solving a problem that doesn't exist yet.

## Consequences

- The glossary gains the term **Module**. The "Skills are the *primary* extensibility mechanism" claim from [ADR-0001](./0001-skill-system-as-primary-extensibility.md) is narrowed: Skills are primary for *capabilities*; Modules are primary for *domains*. The two coexist and a Module's Skills still go through the existing Skill loader.
- A Module's on-disk shape is fixed: `modules/<name>/MODULE.md` (manifest + prose), `schema/*.sql` (versioned migrations applied in order), `src/{service.ts, events.ts, index.ts}` (a `createService(deps)` factory and an `eventTypes` contribution), and `skills/*/SKILL.md` (Skills scoped to this Module). The filesystem is canonical — drop a folder, get a Module.
- Boot lifecycle: filesystem scan → topological sort by `depends_on` → apply migrations tracked in a `schema_migrations` table → instantiate services via `createService({ db, eventBus, storage, otherModules })` → expose handles through a `ModuleRegistry` injected into `ToolContext` and HTTP request context. Failure is **degraded, not fatal** — a broken migration freezes that Module at its last good version and surfaces `module.degraded`; other Modules and the chat UI still boot.
- Open service surface: routes under `/api/<module>/*` call services directly and return JSON without an LLM in the loop. Routes under `/api/sessions/...` keep their existing agent-mediated semantics. The agent becomes one client of the service layer rather than its only client.
- The `audit_log` table is written by Module services on every mutation. It is **not** the same as the event log: the event log is observational and may drop high-frequency events; the audit log is immutable and complete. The two coexist (events for UI/hook signaling; audit for forensic record).
- Module-to-Module communication has two channels: typed service calls (for synchronous needs — `Invoicing.markInvoicedFromProposal(p)`) and event subscriptions (for loose coupling — `invoicing` listens to `proposal.accepted` and seeds a budget snapshot). A Module declares `depends_on` only for the service-call relationship.
- Permission gating remains on the agent's side. `PermissionGate` decides which Modules' Skills the agent can load. HTTP-route callers are treated as the operator and do not pass through this gate — they passed the front door (single-user, local). This keeps the trust model identical to v1.
