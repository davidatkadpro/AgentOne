# Non-chat session activation: spawned sessions, awaiting-input, notifications

A Session can be created programmatically by a Module or HTTP route — not only by a human typing the first message — by supplying a `seed` to `POST /api/sessions`. The orchestrator processes `seed.initialMessage` as the opening user turn and runs the session immediately; provenance is recorded in a new `sessions.spawned_by` column. A Session may pause in a new `awaiting_input` state, reached only by the agent explicitly calling a new `request_user_input(question, options?)` Core Tool — the runtime never promotes a Session to this state implicitly. User-facing surfacing is mediated by a new `notifications` table written by Modules, the orchestrator, and hooks; notifications are dismissable user signals (distinct from the audit log) and the WebSocket emits `notification.*` events for the UI to render a tray. Together these three additions let work begin outside chat, ask for the human's attention when it can't continue alone, and surface that pause in the UI without coupling Modules to chat plumbing.

## Considered alternatives

- **Stay agent-only — non-chat callers route through the agent.** A button click in the email view sends an HTTP request to a fixed prompt template, which becomes a regular chat session. Rejected — forces an LLM call for actions that should be deterministic (filing an email, creating a project), and prevents the agent from being one of several actors. Also blocks the v2 mixed-frontend reality (chat alongside non-LLM panels) accepted in [ADR-0004](./0004-modules-as-second-extensibility-primitive.md).
- **Implicit `awaiting_input` — runtime infers from the agent's last turn.** Reject; if a turn ends with a question, the orchestrator promotes the Session to `awaiting_input`. Rejected because "the agent meant to ask" is unreliable — it reads as flaky to the user and surfaces phantom notifications for ordinary chat turns. The explicit-only model trades a little token cost for a stable contract.
- **Notifications as a single event-bus type, not a table.** UI subscribes; nothing is persisted. Rejected — users need to see unresolved attention items after restart; "I missed a notification because the WebSocket dropped" is the failure mode this avoids.
- **Notifications conflated with the audit log.** One table, one writer policy. Rejected — they answer different questions. The audit log answers "what happened to the system" (forensic, immutable, complete); notifications answer "what does the user need to know about right now" (UI-facing, dismissable, lossy). Conflating them either makes the audit log noisy or makes notifications incomplete.
- **Free-spawn — any Module spawns any Agent Profile.** Rejected — any UI button would effectively bypass the agent-profile permission model. The accepted constraint: a Module's `MODULE.md` declares which Profiles it may spawn, and the operator's settings whitelist further restricts. No new permission subsystem; this stays in module config.

## Consequences

- The glossary gains **Spawned Session**, **Awaiting Input** (Session state), **Notification**, and a fourth complementary term **Audit Log** (introduced by [ADR-0004](./0004-modules-as-second-extensibility-primitive.md) and referenced here).
- `sessions` schema additions: `state TEXT NOT NULL DEFAULT 'active'` (values `'active' | 'awaiting_input' | 'archived'`) and `spawned_by TEXT NULL` (audit field — module path or null for human-created). Existing rows backfill cleanly with defaults.
- One new Core Tool: `request_user_input(question, options?)`. Calling it emits `session.awaiting_input`, sets `state='awaiting_input'`, returns immediately, and ends the turn. Any subsequent user message returns the Session to `'active'`. Options are optional structured choices the UI can render as buttons (e.g. a list of candidate projects).
- `notifications` table: `id`, `kind` (`'info' | 'attention_needed' | 'error'`), `title`, `body`, `session_id?`, `module?`, `payload_json`, `status` (`'unread' | 'read' | 'resolved' | 'dismissed'`), `created_at`, `resolved_at`. Writers: orchestrator (on `awaiting_input`), Modules (on domain events worth surfacing), Hooks (settings-driven). The WebSocket transport gains `notification.created`, `notification.updated`, `notification.resolved`; REST gains `GET /api/notifications`, `PATCH /api/notifications/:id`.
- `POST /api/sessions` accepts an optional `seed: { spawnedBy, initialMessage, context }`. The existing `agentProfile` field remains; profile-spawn whitelisting is enforced at the Module layer before the request is issued. The Path-A profile-mismatch guard (already in place from v1's M14–M19 work) continues to apply.
- The orchestrator's turn loop is unchanged for chat-initiated sessions. The only new path is: at session creation, if `seed.initialMessage` is present, the orchestrator enqueues it as the first user message and runs a turn immediately, emitting `session.spawned` with the `spawned_by` value for the audit log.
- The implicit-input pathway is *not* added even gated on `spawned_by`. If a spawned session ends without `request_user_input`, it stays `active` and quietly waits — the orchestrator does not guess. The Module that spawned the Session may post its own `info` notification ("scope extracted; review and accept") if it wants the user surfaced.
